## Audience Alignment Assessment

See [AUDIENCES.md](../AUDIENCES.md) for full profiles.

### Beginner Alignment: 4/10

**What works:**
- `beetree init` → `beetree new` scaffolding is clean and produces a good template
- Example files (patrol, combat, full-enemy-ai) have excellent inline documentation
- TUI provides visual tree editing without touching YAML directly
- Interactive simulation is conceptually perfect for learning

**What's broken:**
- No onboarding path. A beginner discovers the tool and has no idea what to do first. The website is empty. There's no quickstart guide.
- The TUI launches with no help visible. Key bindings are only in `--help` output, not in the TUI itself.
- Code generation produces files with no integration guidance. A beginner gets 6 C# files and doesn't know what to do with them.
- Error messages assume BT knowledge ("composite must have at least one child" — what's a composite?).
- No concept explainer. The tool uses terms like "selector," "sequence," "blackboard," "decorator" without ever defining them.

**Critical gap:** A beginner cannot go from "I installed this" to "my game has AI" without external documentation that doesn't exist.

### Intermediate Alignment: 6/10

**What works:**
- CLI is comprehensive: validate, render, simulate, diff, generate all work
- TUI has the core editing loop: add/edit/delete/move/undo/save
- Code generation produces correct engine-specific code for all three targets
- Simulation with overrides enables scenario testing from CLI
- Spec format is expressive enough for real game AI

**What's broken:**
- TUI can't edit parameters, blackboard, metadata, or custom_nodes — forces users back to manual YAML for the most common intermediate operations
- No validation feedback in TUI. You save, then separately run `beetree validate`.
- No search/find in TUI. Navigating a 50-node tree is painful.
- Registry is non-functional. Commands exist but backend is a stub.
- Code generation has no migration story. Add a node to your tree → re-generate → but what about existing stubs? When do stubs get orphaned?

**Critical gap:** The TUI claims to be an editor but can only edit 4 of 10+ node fields. Everything else requires YAML.

### Expert Alignment: 3/10

**What works:**
- Spec format covers standard BT semantics well
- CLI is scriptable and composable (validate in CI, generate in build scripts)
- CI pipeline runs vet, staticcheck, race detector, cross-compilation
- Clean Go architecture that's readable and testable

**What's broken:**
- No extension points. Can't add custom generators, custom node types via config, or custom validation rules.
- Registry is non-functional — can't evaluate for team workflow.
- No programmatic API documentation.
- No type mapping customization for code generation (blackboard "Entity" becomes `object`).
- No performance data for large trees.
- Simulation has no test framework — can't write assertion-based tests for tree behavior.

**Critical gap:** An expert evaluating BeeTree for team adoption has no registry, no extensibility, and no team workflow support. They'd pick a different tool.

## Feature Status Assessment

See [FEATURES.md](../FEATURES.md) for full feature list.

### Killer Feature Health

| Feature | Concept | Implementation | UX | Overall |
|---------|:-------:|:--------------:|:--:|:-------:|
| Interactive Simulation | A+ | B | C+ | B- |
| Code Generation | A | B+ | D | C+ |
| TUI Editor | A | B | C | C+ |
| Accessibility | A | C | D | D+ |

**Interactive Simulation** — The concept is the strongest differentiator. Implementation is correct (sequence/selector/parallel/decorator semantics). UX suffers from no help text, no scenario save, no step-back, and no way to pre-set leaf outcomes in TUI mode.

**Code Generation** — Works correctly for all three engines. But the output is opaque — generated files have no integration guide, no README, no comments explaining how to use them. A user gets files and is stuck.

**TUI Editor** — Core editing works (add/edit/delete/move/undo). But it's a half-editor: can't touch parameters, blackboard, metadata, or custom nodes. No search, no validation feedback, no help overlay.

**Accessibility** — The examples are excellent, but they exist in a vacuum. No documentation, no website, no tutorials, no concept explanations. The tool is not accessible to beginners despite that being a stated goal.

## Gap Analysis

### Critical Gaps (blocks adoption)

1. **No documentation.** Website is a 6-line stub. No getting started guide, no CLI reference, no concept explainer, no engine integration guide. This is the #1 blocker for all audiences.

2. **TUI can't edit most node fields.** Parameters, blackboard, metadata, custom_nodes, success/failure policies — none editable in TUI. Users hit the "YAML wall" within 5 minutes.

3. **Code generation has no integration story.** Generated code has no README, no usage comments, no example of how to instantiate and run the tree in each engine.

4. **No in-TUI help.** The TUI is the primary interface but provides no help, no concept explanations, no key binding reference once launched.

### Major Gaps (degrades experience)

5. **No validation in TUI.** Save an invalid tree, discover it's broken only when running `beetree validate` separately.

6. **Registry is a stub.** All registry commands exist but connect to a mock backend. Sharing is the stated #4 value prop and it's non-functional.

7. **Error messages are terse.** "node not found" without suggesting similar names. "composite must have children" without explaining what a composite is.

8. **No quickstart or templates.** `beetree new` always produces the same template. No guided creation, no pattern selection.

### Minor Gaps (polish)

9. **No redo.** Undo exists (50 levels) but no redo.
10. **No search in TUI.** Can't jump to a node by name.
11. **No trace export.** Simulation results can't be saved or compared.
12. **No copy/duplicate node.** Only cut-move, no copy.

## Recommendations — Improvement Plan

### Phase 1: Documentation & Onboarding (highest impact, unblocks beginners)

1. **Write CLAUDE.md** for the repo — establish project conventions
2. **Create `/introspect` slash command** — make checkpoint process repeatable
3. **Add in-TUI help overlay** — press `?` to see key bindings + BT concept quick-reference
4. **Add generated code README** — each `beetree generate` run should produce a README with integration instructions

### Phase 2: TUI Completeness (unblocks intermediates)

5. **Add parameter editing to TUI** — extend the edit form to handle `map[string]interface{}` parameters
6. **Add blackboard editing to TUI** — new modal for adding/editing/removing blackboard variables
7. **Add in-TUI validation** — call validator before save, show errors in properties panel
8. **Add search/jump-to-node** — press `/` to search by name

### Phase 3: Code Generation UX (unblocks all tiers)

9. **Add integration comments to generated stubs** — explain what each stub does and where to implement logic
10. **Add engine-specific integration guide** — one doc per engine showing end-to-end workflow
11. **Template selection for `beetree new`** — choose from patrol/combat/utility/blank

### Phase 4: Expert Features (unblocks adoption)

12. **Scenario test framework** — `beetree test` command that runs named simulation scenarios
13. **Real registry backend** — even a file-based GitHub-backed registry would work
14. **Type mapping configuration** — let users define how blackboard types map to engine types
