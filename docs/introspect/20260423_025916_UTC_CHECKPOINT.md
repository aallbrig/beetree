## Audience Alignment Assessment

### Beginner Alignment: 5/10

**What works**
- `init` + `new` flow is straightforward
- Examples are heavily annotated and pedagogical
- TUI help overlay and search improve self-discovery
- Simulation makes BT flow tangible

**What still hurts**
- No complete docs site onboarding path
- Beginner concept model still split between README/examples/help text
- Some validation/error phrasing remains domain-heavy

**Bottom line:** Better than prior checkpoint, but still not truly beginner-safe end-to-end.

### Intermediate Alignment: 7/10

**What works**
- Strong CLI coverage for core workflows
- TUI editing loop is much more complete than before (params + blackboard + search + redo)
- Generate/validate/simulate loops are productive

**What still hurts**
- Advanced spec facets still require manual YAML
- Registry workflow unavailable in practical terms
- Long-term spec/governance docs are missing for team scaling

**Bottom line:** Day-to-day use is viable, but scale and governance are under-specified.

### Expert Alignment: 4/10

**What works**
- Scriptable CLI and solid internal structure
- Multi-engine generation and test suite give technical credibility

**What still hurts**
- No production registry backend
- No extension contract for generators/validators
- No formal SDD corpus for architecture, semantics, compatibility, and contributor guidance

**Bottom line:** Strong prototype, weak platform governance.

## Gap Analysis

### Critical Gaps

1. **No formal SDD document set.**
   Future contributor sessions must reverse-engineer behavior from code/tests.

2. **Website/docs remain incomplete for onboarding and deep references.**
   Tool capability exceeds published guidance.

3. **Registry remains hidden/stubbed.**
   Team sharing value proposition is still not deliverable.

### Major Gaps

4. **Advanced spec editing parity in TUI is incomplete.**
5. **Simulation lacks saved scenarios and replayable checks.**
6. **No explicit compatibility policy for spec evolution and generated outputs.**

### Minor Gaps

7. **Error taxonomy and message consistency are undocumented.**
8. **Contributor-level architectural boundaries are implicit, not explicit.**

## Proposed SDD Document Set (for future agent sessions)

Create these docs as the initial spec-driven foundation:

1. **`docs/design/sdd_architecture.md`**  
   Module boundaries, responsibilities, dependency direction, ownership rules.

2. **`docs/design/sdd_spec_schema_and_validation.md`**  
   Source-of-truth for `.beetree.yaml` fields, invariants, and validator rules.

3. **`docs/design/sdd_simulation_semantics.md`**  
   Execution semantics by node type/decorator, deterministic assumptions, trace guarantees.

4. **`docs/design/sdd_codegen_contract.md`**  
   Shared generator contract, generated-vs-stub rules, overwrite behavior, naming conventions.

5. **`docs/design/sdd_codegen_unity.md`**, **`sdd_codegen_unreal.md`**, **`sdd_codegen_godot.md`**  
   Engine-specific mapping contracts and integration assumptions.

6. **`docs/design/sdd_tui_interaction_model.md`**  
   Editor states, keybinding behavior, modal transitions, save/validate semantics.

7. **`docs/design/sdd_cli_behavior_contract.md`**  
   CLI command semantics, argument rules, output/error expectations, stability commitments.

8. **`docs/design/sdd_error_taxonomy.md`**  
   Error categories, wording standards, actionable recovery guidance.

9. **`docs/design/sdd_versioning_and_compatibility.md`**  
   Spec versioning policy, deprecation strategy, and backward compatibility expectations.

10. **`docs/design/sdd_contributor_playbook.md`**  
    Required change workflow for agent sessions: update specs first, then implementation/tests.

## Prioritized Improvement Plan

1. **Publish the core SDD spine first:** schema/validation, simulation semantics, codegen contract.  
2. **Lock contributor workflow:** require spec updates for behavior changes.  
3. **Close docs gap:** convert current README/examples into a canonical onboarding path.  
4. **Define compatibility policy before broad adoption.**  
5. **Only then pursue registry productionization and extension APIs.**
